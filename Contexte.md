📋 Présentation généraleL'objectif de ce projet est de concevoir et de développer un système de RAG (Retrieval-Augmented Generation) capable d'analyser un dépôt de code, d'y rechercher les informations pertinentes et de générer des réponses précises à des questions posées par l'utilisateur. Le cas pratique de ce sujet porte sur l'ingestion du dépôt de vLLM.🛠️ Spécifications techniques & EnvironnementLangage : Python 3.10 ou version ultérieure.Gestionnaire de paquets : uv (obligatoire pour la gestion du projet et des dépendances).Qualité du code : Respect strict des normes flake8, utilisation de mypy pour le typage statique, et docstrings conformes à la PEP 257 (style Google ou NumPy).Validation des données : Toutes les classes doivent utiliser Pydantic pour garantir la sécurité des types et la validation.Interface : Une interface en ligne de commande (CLI) construite avec Python Fire, accompagnée de barres de progression tqdm pour les opérations longues.🧩 Fonctionnalités obligatoires (Pipeline RAG)Le projet s'articule autour de 5 composants majeurs à implémenter :1. Ingestion de la base de connaissancesLecture et traitement de tous les fichiers du dépôt vLLM.Mise en place d'une stratégie de découpage (chunking) intelligente différenciée pour le code Python et pour la documentation Markdown.Taille maximale des chunks fixée à 2000 caractères (configurable via la CLI).Création et sauvegarde d'un index de recherche en moins de 5 minutes.2. Système de recherche (Retrieval)Implémentation obligatoire de l'une de ces deux méthodes de recherche textuelle : TF-IDF ou BM25.Retourner les $k$ chunks les plus pertinents sous forme de métadonnées précises : chemin du fichier, index du premier caractère et index du dernier caractère.Doit supporter le traitement par lot (batch) de fichiers de questions au format JSON.3. Génération de réponses (Generation)Utilisation du modèle de langage Qwen/Qwen3-0.6B par défaut.Les réponses doivent être intégrées dans la fenêtre de contexte du LLM (en respectant la limite de tokens) et exportées dans un format JSON structuré par Pydantic.Une bonne réponse doit être autonome, ancrée dans les sources fournies (pas d'hallucination) et pertinente.4. Système d'évaluationCalcul de la métrique Recall@k pour mesurer la qualité de la recherche en la comparant à des annotations de vérité terrain (ground truth).Un élément source est considéré comme "trouvé" s'il y a au moins 5% d'overlap (chevauchement) entre le chunk récupéré et la source correcte.5. Commandes de la CLILe script principal doit exposer les commandes suivantes :index : Indexer le dépôt de code.search : Effectuer une recherche pour une seule requête.search_dataset : Traiter un fichier de questions et exporter les résultats de recherche.answer : Répondre à une seule question en y injectant le contexte.answer_dataset : Générer les réponses de tout un dataset à partir des résultats de recherche.evaluate : Évaluer la qualité de la recherche par rapport à la vérité terrain.⚡ Objectifs de performance attendusLe système doit impérativement respecter les seuils suivants (des scores plus élevés rapportent des bonus) :Temps d'indexation maximal : 5 minutes.Latence au démarrage à froid : 60 secondes maximum (chargement du modèle inclus).Débit de recherche à chaud : Maximum 90 secondes pour traiter 1000 questions.Précision Recall@5 minimale : 80% pour les questions sur la documentation (docs) et 50% pour les questions sur le code (code).🌟 Partie BonusIl est possible d'ajouter des fonctionnalités avancées de RAG pour obtenir des points supplémentaires :Expansion ou réécriture de requêtes (Query expansion).Utilisation d'embeddings sémantiques pour la recherche.Mise en cache des résultats (index ou requêtes).Recherche hybride (combinant recherche textuelle et sémantique).Inférence du LLM en local via vLLM.📦 Soumission et SoutenanceLivrables : Le dépôt Git doit contenir le répertoire src/, les fichiers pyproject.toml, uv.lock, un Makefile (avec les règles install, run, debug, clean, lint et lint-strict) ainsi qu'un fichier README.md complet et rédigé en anglais détaillant l'architecture du système.Note importante : Ne pas inclure de gros fichiers de données, de poids de modèles ou d'outputs générés dans le dépôt.Recode : Pendant la soutenance, l'évaluateur pourra demander une légère modification du code (changement de comportement, ajout d'une fonctionnalité simple en quelques minutes) afin de vérifier la parfaite compréhension du projet.
